How to design a good API and why it matters?

Posted on 2019-05-08 Edited on 2020-12-18

1.前言

1.1.为什么接口设计很重要？

  • API可能是一个公司最重要的资产
    • 用户花了很多心血：购买，使用，学习等
    • 停止使用API的成本可能很高
    • 成功的公开API能够抓住用户
  • API也可能是公司最大的责任之一
    • 糟糕的API会导致无休止的技术支持需求
  • 公开的API是永久性的 - 只有一次机会保证其正确

1.2.为何接口设计对于你很重要？

  • 如果你是程序员，你就是一个API 设计者
    • 好的代码是模块化的 - 每个模块有一个API
  • 有用的模块往往会被重用
    • 一旦模块有用户，则不能随意更改API
    • 良好的可重用模块是企业资产
  • 在API设计上进行思考可提高代码质量

1.3.优秀的API的特点

  • 易于学习
  • 易于使用，即便没有文档
  • 很难被误用
  • 易于阅读和维护使用它的代码
  • 足够强大到满足需求
  • 易于扩展
  • 适合受众

2.概况

2.1.API设计的流程

2.1.1 以合理的怀疑态度收集需求

  • 通常你会得到建议的解决方案
    • 更好的方案可能存在
  • 你的工作是提取出真实的需求
    • 应采用用户案例的形式
  • 构建一个更加通用的API是更简单，且更有价值的

2.1.2 以简短的细则开始 - 1页足够

  • 在这个阶段，敏捷是胜过完整性的
  • 将细则在尽可能多的人之间传播
    • 听取他们的输入，并且慎重对待
  • 如果保持细则简短，则易于修改
  • 持续更新这个细则

2.1.3 尽早，经常写入你的API

  • 在实现之前开始设计API
    • 可以避免会被丢弃的实现
  • 在细则完全确定之前开始设计API
    • 可以避免会被丢弃的细则
  • 持续地更新你的API
    • 避免令人讨厌的惊喜
    • 代码依靠样例和单元测试存活

2.1.4 写入SPI(Service Provider Interface)更为重要

  • Service Provider Interface(SPI) 关于SPI介绍，参见: JavaSPI机制介绍
    • 插件接口，可支持多种实现
  • 在发布前，编写多个插件
    • 如果只写一个插件，可能不能支持另外一个
    • 如果写两个插件，可能支持更多的（需求）
    • 如果写三个插件，可能就能正常工作

2.1.5 维护现实的期望

  • 大多数API设计都过度约束了
    • 你不能使每个人满意
    • 只需要平等的对待每一个人的需求
  • 期望会犯错误
    • 一段时间的真实调用会将这些错误冲刷掉
    • 期待能持续发展API

2.2.设计原则

2.2.1 API应该只做一件事情，并且做好这件事情

  • API的功能应该非常容易解释
    • 如果很难命名API，这可能是一个坏的标志
    • 好的命名驱动着开发
    • 易于拆分和合并模块

2.2.2 API应该尽可能的小

  • API应该满足它的需求
  • 当存在疑问时，放弃这个API
    • 功能，方法，类，参数等
    • 你永远可以添加，但是决不能移除{:.error}
  • 概念比实体更加重要
  • 寻找良好的能效比

2.2.3 实现不应该影响API

  • 实现细节
    • 会迷惑用户
    • 限制改变实现的自由
  • 了解什么是实现细节
    • 不要过度指定方法的行为
    • 例子：不要指定hash函数的细节
    • 所有可调整的参数都是可疑的
  • 不要让实现的细节"泄露"{:.error}到API

2.2.4 最小化所有对象的可访问性

  • 使类和成员变量尽可能是Private的
  • 公有类不应该有公有成员变量（常量除外）
  • 这样可以最大化信息隐藏
  • 允许独立使用，理解，构建，测试和调试模块

2.2.5 命名很重要 - API是一种小语言

  • 名称需要最大化自解释性
    • 避免神秘的缩写
  • 保持一致 - 相同的单词意味着相同的事情
    • 这个原则是横跨所有API的，跨平台的
  • 定期争取对称性
  • 代码需要像散文一样

2.2.6 文档很重要

重用是说起来容易做起来难的事情。实现重用需要好的设计以及非常好的文档。即便我们看到好的设计（这是不常见的），如果没有好的文档，也不会实现目标中的代码重用。

2.2.7 文档的宗旨

  • 为每个类，接口，方法，构造器，参数以及异常编写文档
    • 类：这个类实例代表什么
    • 方法：方法和客户端之间的契约
      • 前置条件，后置条件，副作用
    • 参数：表明单位，类型，所有权
  • 非常仔细地记录状态空间

2.2.8 考虑API设计决策的性能后果

  • 错误的决策会影响性能
    • 使类型是可变的
    • 提供构造器而不是静态工厂，参见：考虑用静态工厂方法代替构造器
    • 使用实现类型而不是接口类型，参见：实现类型vs 接口类型
  • 不要包装API以获得性能
    • 潜在的性能问题可能会被解决，但是造成的后遗症是永久的
    • 好的设计通常拥有优秀的性能

2.2.9 API设计决策对性能的影响是真实和永久的

  • Component.getSize()返回 Dimension
  • Dimension是可变的，每次调用getSize都会分配一个Dimension，所以造成了百万次无用的对象分配
  • 替代版本在1.2添加，但是旧的客户端代码依旧很慢

2.2.10 API必须与平台和平共存

  • 做到习惯做法
    • 遵循标准的命名传统
    • 避免使用过时的参数和返回类型
    • 模仿核心API和语言中的模式
  • 利用对API友好的特性
    • 泛型，变量，枚举，默认参数
  • 了解并避免API陷阱和缺陷
    • Finalizers, public static final arrays

2.3.类设计

2.3.1 最小化可变性

  • 除非有必要，否则类应该是不可变的
    • 优势：简单，线程安全，可重用
    • 劣势：对于每个值都有独立的对象
  • 如果是可变的，保证状态空间尽可能小，定义明确的
    • 明确何时调用哪种方法是合法的

正面案例：TimerTask
反面案例：Date, Calendar

2.3.2 只在有意义时创建子类

  • 子类意味着可替代性
    • 只在存在一对多的(is-a)的关系时运行创建子类
    • 否则，使用组合
  • 公共类不应该为了便于实现，而为其他公共类创建子类

反面案例：Stack extends Vector
Properties extends Hashtable
正面案例：Set extends Collection

2.3.3 对于继承需要设计和文档，否则就禁止

  • 继承违反了封装原则
    • 子类对于父类的实现细节是敏感的
  • 如果你允许创建子类，则文档自用
    • 方法如何相互使用
  • 保守政策：所有的具体类都是final的 参见: 深入理解Java中的final关键字

反面案例：Many concrete classes in J2SE libraries
正面案例：AbstractSet, AbstractMap

2.4.4 慎重地进行重载

  • 避免模糊的重载行为
    • 对于同一个实际行为写了多个重载
    • 保守原则：相同数目的参数不要有两个（重载方法）
  • 仅仅因为你可以并不代表你应该这样做
    • 这个时候一般使用另一个名称会更好
  • 如果你提供了模糊的重载，保证相同的参数产生相同的行为

2.4.5 使用合适的参数类型和返回类型

  • 对于输入，相较与类，鼓励使用接口类型
    • 提供了灵活性，性能
  • 使用最具体的输入参数类型
    • 将错误从运行时移到了编译时
  • 如果存在更合适的类型，不要使用String
    • String是笨重的，易出错的，慢的
  • 不要使用float类型用于货币价值
    • 双浮点数容易造成不准确的结果
  • 使用double(64 bits)而不是float(32 bit)
    • 精度丢失是真实的，而性能并没有提升多少

  • 三个或更少的参数比较理想

    • 程序员可能错误地转置参数
    • 程序可能会正常的编译，运行，但是行为错误了

  • Checked - 客户端必须采用恢复措施
  • Unchecked - 程序错误
  • 过度使用受检异常会导致样板代码(boilerplate)

  • 便于诊断和恢复
  • 对于未受检异常，消息就足够了
  • 对于受检异常，请提供访问者信息

  • 没有文档的话，很难使用

  • 没有文档，也能简单使用

3.总结

  • API设计是一个严肃且有回报的工艺
    • 改善了很多编程者，终端用户，公司等
  • 这篇文档涵盖了一些该工艺的启发式方法
    • 不要无脑附和
    • 在没有合理的原因的情况下，也不要违背
  • API设计很难
    • 它不是一个单独的过程
    • 完美是不存在的，但是可以持续改进
